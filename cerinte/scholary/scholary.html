1. "Introduction"

1.1 "Purpose"

The purpose of this document is to present a detailed description of the web application Skill Virtual Assistant(SkiVI). It will explain the purpose and features of the application, the interfaces, what it will do and how it operates. This document is intended for users of the application and potential developers.

SkiVi is a modular Web application with the role of a virtual instructor that offers sets of activities and instructional resources for learning/developing some skills such as playing an instrument, ressuscitaion procedures, origami and others.

Each skill will be implemented using an independent Web microservice that will be updated on demand or automatically.

The base functionalities of the system will be offered by a REST or GraphQL API.

2. "Overall Description"

2.1 "Product Perspective"

SkiVI was developed for everyone who is interested in learning new hobbies, skills, informations, facts,etc. who wants either to just have some relaxing time, or to properly learn something useful that can be put into reality.

It can provide feedback to its users, get feedback from them, and was developed to run on web browsers.

2.2 "Product Functions"

Main Pages:

  • Home: The home page of the application, it contains basic informations about it and some courses reccomended by the developers.
  • Courses: The main page of the application, it contains the courses and lessons available, it also contains a search function for them using keywords or labels.
  • Course page: A page that appears only when an user selects a course or lesson, in which he can see its description, details, and its contents.
  • Propose a course: A form page in which users can propose a course or lesson to be added in the application.
  • Feedback: A form page in which users can give feedback to the application
  • Log In: A log-in or register page in which users can log into their accounts or make new accounts.

2.3 "User Classes and Characteristics"

  • Typical Users (students, teachers, anyone mostly), who want to use SkiVI for learning.
  • Advanced/Professional Users (experts, researchers), who want to use SkiVI to broaden its contents.
  • Programmers who are interested in working on the project by further developing it or fixing bugs.

2.4 "Operating Environment"

Any Operating System with GUI that has access to an web browser.

2.5 "Design and Implementation Constraints

The website uses HTML:5, JavaScript, PHP and MySQL for presenting the informations, storing user data, sending emails, storing and accessing infromation about lessons.

2.6 "User Documentation"

The users can use this application using XAMPP, by making the folder in which the SkiVI folder is on their file system the document root of the Apache server.

In order for the application to properly send emails on feedback and course proposal, please reffer to the email.txt file present in the same folder in which this document's parent folder resides.

After that the user can start the MySQL module of XAMPP and start using the application

In order to view the homepage, the user shall open the public folder in the localhost page

After that the user will be able to see the main page, check what courses are available, and login or register

In order to view courses' contents, leave feedback or propose a course, the users must login with an existing account

The users(those that have access to the microservices api) can also update, create or delete lessons using Postman and proper calls to the microservices.

2.7 "Assumtions and Dependencies"

SkiVI uses some JavaScript therefore requires browsers that support it.

4. "System Features"

This section presents SkiVI's most prominent features and explains how they can be used and the results they will give back to the user.

4.1 "Login/Register"

Unlogged Users can use this function in order to create or connect to their accounts, that will keep track of their progress.

The application will store the users credentials in a MySQL database for easier access to their accounts.

4.2 "Learn a course"

Logged Users can browse the application's courses and lessons and choose one that interests them.

After choosing one, they are sent to it's own page that contains details and their contents where they will learn the skills.

(Optional) We will try to implement a function to keep track of their progress.

4.3 "Propose a course"

Logged Users can propose a course or lesson to be added in the application.

After completing the form they will receive via e-mail our response about their proposal with further instructions to follow.

4.4 "Give Feedback"

Logged Users can give feedback for our web application about it's general view, what they like/dislike about it, and anything else they would like to share.

4.5"Microservices"

SkiVI uses microservices in order to receive the lessons and lessons' contents from the database. The connection with microservices is made with an AJAX call using GET request method to the microservice's api part that deals with giving the informations of all lessons of a given course.

Those microservices also can receive POST, PUT, and DELETE requests in order to update, create or delete lessons, for more informations please check the microservices API document.

5. "Other Nonfunctional Requirements"

5.1 "Performance Requirements"

SkiVI requires a system that has access to Web browsers and that can display images, videos etc. Performance depends on system RAM, browser CPU & RAM consumption.

5.2 "Safety Requirements"

Developers will try to update SkiVI constantly based on user's feedback. If there is a bug or error, users can use Feedback function to report it.

5.3 "Security Requirements"

SkiVI does not have any security requirements so any type of user can use it wihour any additional privileges.

5.4 "Software Quality Attributes"

SkiVI provides the users with intuitive features. Due to its design, it is easy to use the interface, derive the function of each page, and doesn't overload the user with content.

5.5 "Business Rules"

SkiVI will not participate in any business related matters.
